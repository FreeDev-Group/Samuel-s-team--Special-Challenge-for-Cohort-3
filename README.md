# MKB Cohort 3 - Survey App Testing Hub

Welcome to the Cohort 3 testing phase of the Student Survey Application!  
This repository serves as the central hub for our collaborative Quality Assurance (QA) and testing process.  
Instead of building the project from scratch, Cohort 3 will put on their "Tester" hats.  
Your mission is to verify whether the application successfully meets the initial use cases defined during its design phase.

## The Testing Environment
All tests are to be conducted on the live production staging site:  
**[https://student.michaelkentburns.com/](https://student.michaelkentburns.com/)**

## Purpose of this repository
We have successfully built the core features of the survey application. Now, we need to ensure it is bulletproof and works exactly as described in our system specifications.  
The `/doc` folder in this repository contains the official **Use Cases** for the software. As a mentee, you will be assigned to specific use cases. You will test the live site, verify that the application handles the scenarios perfectly, and document any bugs or edge cases you discover.

## Testing workflow
Our project will proceed in the following stages:

1. **Claim an Issue:** The mentor will create step-by-step issues in the **Issues** tab representing specific use cases or user flows. You will be assigned to or will claim an issue.
2. **Understand the Use Case:** Navigate to the `doc/` folder in this repository and thoroughly read the use cases provided.
3. **Execute the Test:** Go to the live testing site and try to execute the "Typical course of events" described in your assigned use case.
4. **Document & Report:** You will document your verification with notes and screenshots. If something fails or doesn't look right, you will report it in the ISSUE thread.

## Collaboration guidelines
To simulate a professional software testing environment, we will follow these strict rules:

1. **Everything Starts with an Issue:** You should only test or document items that have a dedicated issue in this repository created by the mentor.
2. **Issue Referencing:** If you push any documentation or write-ups to this repository, your commit message must identify the issue by number (e.g., `test: Verified manage survey use case #4`).
3. **Communication:** Use the **GitHub Discussions** tab or the comment section of your ASSIGNED ISSUE to discuss edge cases, weird behaviors, or questions with your mentor.
4. **Testing Progress:** Mentees are expected to report their testing progress in their respective ISSUE threads.
